# Архитектура субагентов UraanxAI

> Многоуровневый пайплайн разработки с проверкой безопасности

## Команда субагентов

### 🔍 РАЗВЕДКА (Explore Agent)
**Роль:** Исследование кодовой базы, поиск паттернов, анализ зависимостей
**Когда:** Перед любой задачей — сначала понять что есть, потом менять
**Количество:** До 3 параллельно (каждый — своя зона ответственности)
**Пример:**
- Agent 1 → ищет все файлы затронутые задачей
- Agent 2 → ищет похожие паттерны/утилиты для переиспользования
- Agent 3 → проверяет тесты и зависимости

### 📐 АРХИТЕКТОР (Plan Agent)
**Роль:** Проектирование решения, выбор подхода, оценка рисков
**Когда:** Перед реализацией нетривиальных задач
**Принцип:** Сначала план → одобрение → только потом код

### 🔨 РАЗРАБОТЧИКИ (General Agent × до 5)
**Роль:** Написание кода, реализация фичей, фикс багов
**Когда:** После одобрения плана
**Количество:** До 5 параллельно для независимых задач
**Принцип:** Работают в изолированных worktree (не трогают main)
**Распределение:**
- Frontend Dev (webapp/) — React, Tailwind, Telegram Mini App
- Backend Dev (server/) — Express, PostgreSQL, API
- Bot Dev (bot/) — grammY, пуш-система
- Infra Dev — деплой, CI/CD, оптимизация
- Fullstack Dev — сквозные задачи, интеграция

### 🛡️ СБ — СЛУЖБА БЕЗОПАСНОСТИ (Security Review Agent)
**Роль:** Проверка кода на уязвимости ПЕРЕД мержем
**Когда:** После каждого изменения кода, ДО деплоя
**Чеклист:**
- [ ] Нет утечки секретов (API keys, tokens, passwords в коде)
- [ ] Нет SQL-инъекций (параметризованные запросы)
- [ ] Нет XSS (экранирование пользовательского ввода)
- [ ] Нет command injection (нет eval, нет shell exec с юзер-инпутом)
- [ ] CORS настроен правильно
- [ ] JWT проверяется на всех защищённых роутах
- [ ] Нет hardcoded credentials
- [ ] Template literals не ломают inline JS (проверка через Node.js)
- [ ] Нет open redirect
- [ ] Rate limiting на критичных эндпоинтах

### ✅ РЕВЬЮЕР (Code Review Agent)
**Роль:** Проверка качества кода ПОСЛЕ СБ
**Когда:** После прохождения СБ-проверки
**Чеклист:**
- [ ] Код решает именно ту задачу, которую просил пользователь
- [ ] Нет лишних изменений (не трогает то, что не просили)
- [ ] Edge cases обработаны (пустые данные, ошибки сети, таймауты)
- [ ] Совместимость с целевыми устройствами (iOS/Android/Telegram)
- [ ] Нет дублирования кода
- [ ] Нет сломанных импортов

### 🧪 ТЕСТИРОВЩИК (Validation)
**Роль:** Автоматическая проверка перед деплоем
**Когда:** После Code Review
**Обязательные проверки:**
```bash
# TypeScript — нет ошибок типов
cd webapp && npx tsc --noEmit
cd server && npx tsc --noEmit

# Билд — собирается без ошибок
cd webapp && npm run build

# Inline JS — нет syntax errors (для файлов с template literals)
node -e "new Function(extractedJS)"

# API endpoints — отвечают корректно
curl -s https://sakhaai-production.up.railway.app/health
curl -s -X POST .../panel/login -d '{"password":"..."}'
```

## Пайплайн (порядок выполнения)

```
┌─────────────────────────────────────────────────┐
│  1. ЗАДАЧА от пользователя                       │
├─────────────────────────────────────────────────┤
│  2. РАЗВЕДКА (Explore × 1-3)                     │
│     → Понять текущее состояние кода              │
├─────────────────────────────────────────────────┤
│  3. АРХИТЕКТОР (Plan × 1)                        │
│     → Спроектировать решение                     │
│     → Одобрение пользователем                    │
├─────────────────────────────────────────────────┤
│  4. РАЗРАБОТКА (General × 1-3, параллельно)      │
│     → Написать код в изолированном worktree      │
├─────────────────────────────────────────────────┤
│  5. СБ-ПРОВЕРКА (Security Agent)                 │
│     → OWASP, secrets, injection, template bugs   │
│     → ❌ БЛОКИРУЕТ деплой при находках           │
├─────────────────────────────────────────────────┤
│  6. CODE REVIEW (Review Agent)                   │
│     → Логика, edge cases, совместимость          │
│     → ❌ БЛОКИРУЕТ деплой при находках           │
├─────────────────────────────────────────────────┤
│  7. ТЕСТЫ (TypeScript + Build + Syntax + curl)   │
│     → ❌ БЛОКИРУЕТ деплой при ошибках            │
├─────────────────────────────────────────────────┤
│  8. ОТЧЁТ пользователю                           │
│     → Что сделано, что проверено, риски           │
│     → Одобрение пользователем                    │
├─────────────────────────────────────────────────┤
│  9. ДЕПЛОЙ (git push → Railway auto-deploy)      │
│     → Ждём завершения деплоя                     │
│     → curl health check                          │
├─────────────────────────────────────────────────┤
│ 10. ВЕРИФИКАЦИЯ на проде                         │
│     → curl критичных endpoints                   │
│     → Отчёт пользователю                         │
└─────────────────────────────────────────────────┘
```

## Правила

1. **Никогда не деплоить без прохождения шагов 5-7**
2. **При провале любого шага — остановка, диагностика, повторная разработка**
3. **Параллельность**: шаги 2 и 4 могут использовать до 5 агентов параллельно
4. **Изоляция**: разработка в worktree, не в main напрямую
5. **Прозрачность**: каждый шаг оставляет отчёт для пользователя
6. **Минимум участия пользователя**: дёргать Игоря только для одобрения плана и финального деплоя. Все промежуточные решения принимать самостоятельно
7. **Автообновление**: если агент не справляется — заменить подходом или переназначить задачу
8. **Проверка кода обязательна**: после написания кода ВСЕГДА запускать СБ + Ревьюера параллельно, не пропускать
